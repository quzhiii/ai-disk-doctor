# Windows AI Space Manager v1.0.0

## Summary

This release packages the first complete local workflow for Windows AI space diagnosis and conservative cleanup orchestration.

The CLI now covers `scan`, `plan`, `clean`, `restore`, `doctor`, `diff --latest`, and `--rules-repo` community rule loading. The default posture remains conservative: diagnose first, explain risks, prefer dry-run, and use quarantine for any real movement.

## Included Workflows

- `scan`: rule-driven discovery, top findings, risk summaries, volume summaries, and scan snapshot persistence.
- `plan`: safe-only cleanup candidates, sensitive-path blocking, and recently-modified filtering.
- `clean`: dry-run reports and quarantine execution with recovery indexes.
- `restore`: dry-run and execution from quarantine indexes, with conflict protection.
- `doctor`: targeted Docker, WSL, Ollama, Hugging Face, and Playwright diagnostics.
- `diff --latest`: compare the newest two scan snapshots without manually selecting files.
- `--rules-repo`: load local or HTTPS git community rules, preferring `<repo>/rules` when present.

## Safety Boundaries

- `clean` execution requires explicit `--yes`; the release smoke script never uses it.
- Real cleanup uses quarantine rather than direct deletion.
- `restore` does not overwrite existing destination paths and reports `skipped-conflict`.
- Sensitive paths are blocked by policy before cleanup candidate generation.
- `--rules-repo` rejects `http://`, `file://`, localhost, private network, and metadata-style endpoints.

## Test Plan

Run these from the repository root or from `aidisk` as noted:

```powershell
cd aidisk
cargo test
..\scripts\release-smoke.ps1
```

The smoke script exercises `scan`, `plan`, `clean --dry-run`, `doctor`, explicit `diff --before`, and local `--rules-repo` loading. It does not perform real quarantine execution.

## Known Limits

- Remote `--rules-repo` support clones HTTPS git repositories but does not auto-update cached clones yet.
- `diff --latest` requires at least two scan snapshots.
- `clean --yes` is intentionally conservative and still depends on filesystem lock state.
- WSL/Docker compaction remains guided through `doctor`; the tool does not directly compact virtual disks.
- Rule contribution governance is documented only at a basic workflow level for this release.
